## 스프링의 정의 및 사용 이유

스프링의 단어는 문맥에따라..
스프링 DI 컨테이너 기술, 스프링 프레임워크, 스프링의 모든 생태계

- 핵심 기술: DI, AOP, IoC, 이벤트
- 웹 기술: 스프링 MVC, 스프링 WebFlux
- 데이터 접근 기술: 트랜잭션, JDBC, ORM, XML
- 기술 통합: 캐시, 이메일, 원격접근, 스케쥴링


스프링의 핵심 개념
아무리 복잡한 기술이더라도
- 이 기술을 왜 만들었는가?
- 이 기술의 핵심 컨셉은?

스프링은 자바 언어 기반의 프레임웤, 자바는 객체지향언어
스프링은 객체 지향언어가 가진 강력한 특징을 살려내는 프레임웤
스프링의 본질: 스프링은 좋은 객체 지향 애플리케이션을 개발할 수 있도록 도와주는 프레임웤
EJB를 쓰면 객체 지향이 무시되고 EJB에 종속적이게됨


## 좋은 객체 지향?

객체 지향 프로그래밍은 컴퓨터 프로그램을 명령어의 목록으로 보는 시각에서 벗어나 여러개의 독립된 단위, 즉 '객체'들의 무임으로 파악하고자 하는 것이다. 각각의 객체는 메시지를 주고받고, 데이터를 처리할 수 있다.(협력)

객체 지향 프로그래밍은 프로그램을 유연하고 변경이 용이하게 만들기 때문에 대규모 소프트웨어 개발에 많이 사용된다.

유연하고 변경이 용이?
- 레고 블럭 조립하듯이
- 키보드 마우스 갈아 끼우듯이
- 컴퓨터 부품 갈아 끼우듯이
- 컴포넌트를 쉽고 유연하게 변경하면서 개발할 수 있는 방법

다형성
- 궁극의 유연함, 궁극의 변경 용이성
- 다형성을 실세계로 비유하면, 역할과 구현으로 세상을 구분

- 운전자, 자동차
    - 운전자 역할, 자동차 역할, 자동차에 대한 구현은 K3, 테슬라차, 아반떼, ...
    - 운전자는 K3에서 아반떼로 차를 바꿔도 운전할 수 있음
    - 자동차가 바뀌어도 운전자한테 영향을 안줌
    - 자동차의 역할을 만들고 구현을 분리한 것은 운전자를 위함
    - 구현 내부가 바뀌어도 역할이 그대로면 외부(클라이언트)에게 영향을 주지 않음
    - 클라이언트에게 영향을 주지 않고도 구현을 계속해서 확장가능

- 공연, 무대
    - 로미오 줄리엣 공연
    - 로미오 역할과 줄리엣 역할 존재
    - 로미오 역할을 장동건이 할 수도, 원빈이 할 수도
    - 줄리엣 역할을 김태희가 할 수도, 송혜교가 할 수도
    - 역할에 따른 연기자를 대체할 수 있음

즉, 세상을 역할과 구현을 분리
- 역할과 구현으로 구분하면 세상이 단순해지고 유연해지며 변경도 편리해진다.
    - 클라이언트는 역할만 알면된다.
    - 클라이언트는 구현 대상의 **내부 구조를 몰라도** 된다.
    - 클라이언트는 구현 대상의 **내부 구조가 변경되어도** 영향을 받지 않는다
    - 클라이언트는 **구현 대상 자체를 변경해도** 영향 받지 않는다.
- 자바 언어의 다형성을 활용
    - 역할 = 인터페이스
    - 구현 - 인터페이스를 구현한 클래스, 구현 객체
    - 객체를 설계할때 역할과 구현을 명확히 분리
    - 객체 설계시 역할(인터페이스)을 먼저 부여하고 그 역할을 수행하는 구현 객체 만들기
        - 핵심은 구현보다 인터페이스가 우선이다
        - 인터페이스를 안정적으로 잘 설계하는 것이 중요

    - 혼자 있는 객체는 없다.
        - 수 많은 객체 클라이언트와 객체 서버(컴퓨터 서버말하는거 아님)는 서로 협력 관계를 가진다.


- 다형성의 본질
    - 인터페이스르 구현한 객체 인스턴스를 실행 시점에 유연하게 변경할 수 있다.
    - 다형성의 본질을 이해하려면 협력이라는 객체 사이의 관계에서 시작해야함
    - 클라이언트를 변경하지 않고 서버의 구현 기능을 유연하게 변경할 수 있다.


- 역할과 구현을 분리
    - 실세계의 역할과 구현이라는 편리한 컨셉을 다형성을 통해 객체 세상으로 가져올 수 있음
        - 인터페이스를 안정적으로 설계하는 것이 정말 중요
    - 역할 자체가 변하면, 클라이언트, 서버 모두에 큰 변경이 발생, 다 뜯어 고쳐야함
        - 인터페이스를 안정적으로 설계하는 것이 정말 중요
            - 최대한 변화가 없는 방식으로

- 스프링과 객체 지향
    - 객체 지향 4대 특징 중 다형성이 가장 중요하다
    - 스프링은 다형성을 극대화 해서 이용할 수 있게 도와준다.
    - IoC, DI는 다형성을 활용해서 역할과 구현을 편리하게 다룰 수 있도록 지원한다.
    - 스프링을 사용하면 레고 블럭 조립하듯이, 공연 무대의 배우를 선택하듯이 구현을 편리하게 변경할 수 있다.


- 스프링을 제대로 이해하려면 SOLID또한 이해해야함
    - 로버트 마틴이 좋은 객체 지향 설계의 5개 원칙을 정리
        - SRP
        - OCP
        - LSP
        - ISP
        - DIP

- SRP
    - 한 클래스는 하나의 책임만 가져야한다.
        - 하나의 책임이라는 것은 실무에서 모호함.
            - 큰책임, 상대적으로 작은 책임이 존재
            - 책임을 너무 크게 잡으면 책임이 많아지고, 책임을 너무 잡게 잡으면 너무 잘게 나눠짐
            - 이것들을 적절히 조절하는 것이 객체 지향의 묘미이다.
        - 중요한 기준은 변경이다.
            - 변경이 있을때 하나의 지점이나 하나의 클래스만 고치면 단일책임의 원칙을 잘따른것

- OCP
    - 확장에는 열려 있고 변경에는 닫혀 있어야한다.
        - 어떻게 코드의 변경없이 기능을 추가할 수 있는거지?
        - 다형성을 활용해서 OCP를 따를 수
            - 인터페이스를 구현한 새로운 클래스를 하나 만들어서 새로운 기능을 구현
    - OCP의 문제점
        ```java
        private class MemberService {
            MemberRepository m = new MemoryMemberRepository(); // 기존코드
        }

        private class MemberService {
            MemberRepository m = new JdbcMemberRepository(); // 변경 코드
        }
        ```
        - 구현 객체를 변경하려면 클라이언트 코드를 변경해야 한다.
            - 분명 다형성을 사용했지만 OCP 원칙을 지킬 수 없다.
        - 객체를 생성하고 연관관계를 맺어주는 별도의 조립, 설정자가 필요하다.
            - 스프링 컨테이너가 수행
    - OCP를 하기위해 DI, IoC가 필요

- LSP
    - 프로그램의 객체는 프로그램의 정확성을 깨뜨리지 않으면서 하위 타입의 인스턴스로 바꿀 수 있어야한다.
    - 즉, 상위 객체를 하위 객체로 변경했을 때에도 규약대로 동작해야한다.
    - ex) 자동차 인터페이스의 엑셀은 앞으로 가라는 기능, 뒤로 가게 구현하면 LSP 위반

- ISP
    - 특정 클라이언트를 위한 인터페이스 여러 개가 범용 인터페이스 하나보다 낫다.
    - 자동차 인터페이스가 너무 크니까 운전 인터페이스, 정비 인터페이스로 분리
    - 사용자 클라이언트를 운전자 클라이언트, 정비사 클라이언트로 분리
    - 분리하면 정비 인터페이스 자체가 변해도 운전자 클라이언트에 영향을 주지 않음
    - 인터페이스가 명확해지고 대체 가능성이 높아진다.

- DIP
    - 추상화에 의존해야지, 구체화에 의존하면 안된다.
    - 즉, 상위 모듈은 하위 모듈에 의존하면 안된다.
    - 구현 클래스에 의존하지말고 인터페이스에 의존하라는 것
    - 역할에 의존하게 해야하는 것 = DIP
        - 구현에 의존하면 변경이 아주 어려워진다.
    - 시스템도 언제든지 갈아끼울수 있도록 만들어야함.
    ```java
        private class MemberService {
            MemberRepository m = new MemoryMemberRepository(); // 기존코드
        }
    ```
    - 위의 코드에서 MemberService는 MemberRepository 인터페이스 에 의존하고 있지만 구현 클래스(MemoryMemberRepository)도 동시에 의존하고 있는 것
    - MemberService 클라이언트가 구현 클래스를 직접 선택
    - 이는 DIP를 위반하는 것

- OCP랑 DIP가 제일 중요하다
    - 객체 지향의 핵심은 다형성
    - 다형성 만으로는 쉽게 부품을 갈아 끼우듯이 개발할 수 없다.
    - 다형성 만으로는 구현 객체를 변경할 때 클라이언트 코드도 함께 변경된다.
    - 다형성 만으로는 OCP, DIP를 지킬 수 없다.
    - 뭔가 더 필요하다.



## 스프링
- 스프링은 다음 기술로 다형성 + OCP, DIP를 가능하게 지원
    - DI 컨테이너
    - DI
- 클라이언트의 코드 변경 없이 기능확장
- 쉽게 부품 교체하듯이 개발

## 스프링이 없던 시절
- 순수하게 자바로 OCP나 DIP 원칙을 지키면서 개발을 하려니 너무 할일이 많았음. 배보다 배꼽이 더 크다. 그래서 프레임워크로 만들어버림

### 정리
- 역할과 구현을 분리할 때 이상적으로는 모든 설계에 인터페이스를 부여하자.
    - 실무적에서 고민
        - 하지만 인터페이스를 도입하면 추상화라는 비용이 발생
            - 추상화가 되면 개발자 코드를 한번 더 열어봐야함
                - 코드 까봤을 때 인터페이스만 보임, 구현 클래스가 뭐지?
                - 코드가 추상화 됨으로써 오는 단점도 존재
                - 항상 장점이 단점을 넘어설때 선택해야함
                
        - 기능을 확장할 가능성이 없다면, 구체 클래스를 직접 사용하고, 향후 꼭 필요할 때 리팩터링해서 인터페이스를 도입하는 것도 방법
        - 이런것을 잘 하는 것이 경험이 많고 잘하는 아키텍트
            - 사실 개발자랑 아키텍트를 분리하는 것도 애매함

    - 객체 지향의 사실과 오해
    - 토비의 스프링 (필수)
        - 스프링을 쓴다면 완독해야함
    - 자바 ORM 표준 JPA 프로그래밍
